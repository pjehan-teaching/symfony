Événements

Le pattern Observer

Symfony repose sur un EventDispatcher : à des moments clés, un composant émet un événement, et d'autres parties du code peuvent y réagir sans que l'émetteur les connaisse.

Cela permet de découpler le code : ajouter un comportement se fait sans modifier l'existant.

On distingue :

  • Un Listener : réagit à un seul événement (via un attribut)
  • Un Subscriber : déclare lui-même la liste des événements qu'il écoute
EventDispatcher

Les événements du Kernel

Lors du traitement d'une requête, Symfony émet une série d'événements dans l'ordre :

Événement Classe Moment
kernel.request RequestEvent Au tout début (avant le routing)
kernel.controller ControllerEvent Contrôleur résolu, avant son exécution
kernel.view ViewEvent Le contrôleur n'a pas renvoyé de Response
kernel.response ResponseEvent Avant l'envoi de la réponse
kernel.exception ExceptionEvent Une exception a été levée
kernel.terminate TerminateEvent Après l'envoi de la réponse au client

Créer un Listener

Un Listener est une classe avec l'attribut #[AsEventListener]. Ici, on ajoute un en-tête HTTP personnalisé à chaque réponse : 

namespace App\EventListener;

use Symfony\Component\EventDispatcher\Attribute\AsEventListener;
use Symfony\Component\HttpKernel\Event\ResponseEvent;

#[AsEventListener]
final class ResponseHeaderListener
{
    public function __invoke(ResponseEvent $event): void
    {
        if (!$event->isMainRequest()) {
            return; // Ignorer les sous-requêtes
        }

        $event->getResponse()->headers->set('X-Powered-By', 'Symfony');
    }
}

Grâce à l'autowiring, le Listener est automatiquement enregistré : aucune configuration requise.

Créer un Subscriber

Un Subscriber implémente EventSubscriberInterface et déclare ses événements :

namespace App\EventSubscriber;

use Symfony\Component\EventDispatcher\EventSubscriberInterface;
use Symfony\Component\HttpKernel\Event\ExceptionEvent;
use Symfony\Component\HttpKernel\KernelEvents;

class ExceptionSubscriber implements EventSubscriberInterface
{
    public static function getSubscribedEvents(): array
    {
        return [
            KernelEvents::EXCEPTION => 'onKernelException',
        ];
    }

    public function onKernelException(ExceptionEvent $event): void
    {
        $exception = $event->getThrowable();
        // Ex : journaliser l'erreur ou renvoyer une réponse JSON pour une API
    }
}

💡 php bin/console debug:event-dispatcher liste tous les listeners par événement.

Événements personnalisés

On peut aussi créer ses propres événements métier. D'abord la classe d'événement :

namespace App\Event;

use App\Entity\Order;
use Symfony\Contracts\EventDispatcher\Event;

class OrderPlacedEvent extends Event
{
    public function __construct(
        public readonly Order $order
    ) {
    }
}

Puis on le dispatche via EventDispatcherInterface :

public function checkout(EventDispatcherInterface $dispatcher, Order $order): Response
{
    // ... enregistrer la commande

    $dispatcher->dispatch(new OrderPlacedEvent($order));
    // Un listener pourra alors envoyer l'e-mail, mettre à jour un stock, etc.
}

Listener ou Subscriber ?

Listener Subscriber
Déclaration Attribut #[AsEventListener] Méthode getSubscribedEvents()
Où voir les événements écoutés ? Dans la configuration / l'attribut Dans la classe elle-même
Idéal pour Réagir à un seul événement Regrouper une logique liée à plusieurs événements

Les deux sont fonctionnellement équivalents : le Subscriber est simplement plus autonome et réutilisable.
