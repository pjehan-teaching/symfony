Le modèle

L'ORM : Doctrine

Symfony recommande l’utilisation de l’ORM (Object-Relational Mapping) Doctrine.

Ce dernier sera en charge de toute la partie “Model” de notre application (génération des entités, persistance des données, requêtes...).

Doctrine est un projet indépendant de Symfony, intégré via le DoctrineBundle.

Symfony Doctrine

Installation

composer require doctrine
php bin/console list doctrine # Voir la liste des commandes ajoutées par Doctrine

Configuration

  • .env / .env.local : Définir les paramètres de connexion à la BDD (variable DATABASE_URL)
  • config/packages/doctrine.yaml : Configuration du driver et du mapping

Création

Database first vs Model first vs Code first

Il existe plusieurs approches pour gérer la partie “Model” d’une application :

  • Database First : BDD → Model → Entités
  • Model First : Model → Code → BDD
  • Code First : Code → Model → BDD

Dans Symfony, le Model et le Code sont fortement liés, surtout avec l’utilisation des attributs.

La méthode recommandée est celle du Code First.

Database First

Les commandes de reverse engineering (doctrine:mapping:import, doctrine:generate:entities) permettaient de générer les entités à partir d'une base de données existante.

⚠️ Elles ont été supprimées de Doctrine ORM 3. Doctrine ne générait qu'environ 70 à 80 % du code : les associations inversées, l'héritage ou les suppressions en cascade devaient de toute façon être écrits à la main.

L'approche Code First est donc aujourd'hui la seule officiellement supportée. Pour une base existante, il faut écrire les entités manuellement, puis vérifier la cohérence du mapping avec : 

php bin/console doctrine:schema:validate

Code First

Création d’une nouvelle entité :

php bin/console make:entity Product
  • Génération des accesseurs (getter / setter)
  • Génération de la classe Repository
  • Génération du constructeur

Code First

Génération du fichier SQL de migration :

php bin/console make:migration

Mise à jour de la base de données :

php bin/console doctrine:migrations:migrate

Créer les associations

Que l’on utilise la méthode Database First ou Code First, il est toujours nécessaire d’écrire les associations à la main.

Une page de la documentation de Doctrine répertorie les différents types d’associations disponibles.

Associations

One-To-Many

use Doctrine\Common\Collections\ArrayCollection;

#[ORM\Entity]
class Product
{
    /**
     * One Product has Many Features.
     */
    #[ORM\OneToMany(targetEntity: Feature::class, mappedBy: "product")]
    private Collection $features;

    public function __construct() {
        $this->features = new ArrayCollection();
    }
}
#[ORM\Entity]
class Feature
{
    /**
     * Many Features have One Product.
     */
     #[ORM\ManyToOne(targetEntity: Product::class, inversedBy: "features")]
     #[ORM\JoinColumn(name: "product_id", referencedColumnName: "id")]
    private ?Product $product = null;
}

Many-To-Many

Owning side

use Doctrine\Common\Collections\ArrayCollection;

#[ORM\Entity]
class User
{
    /**
     * Many Users have Many Groups.
     */
    #[ORM\ManyToMany(targetEntity: Group::class, inversedBy: "users")]
    #[ORM\JoinTable(name: "users_groups")]
    private Collection $groups;

    public function __construct() {
        $this->groups = new ArrayCollection();
    }
}

Inverse side

use Doctrine\Common\Collections\ArrayCollection;

#[ORM\Entity]
class Group
{
    /**
     * Many Groups have Many Users.
     */
    #[ORM\ManyToMany(targetEntity: User::class, mappedBy: "groups")]
    private Collection $users;

    public function __construct() {
        $this->users = new ArrayCollection();
    }
}

Cascade et orphanRemoval

Par défaut, Doctrine ne persiste ou ne supprime que l'entité sur laquelle on agit. Les options de cascade permettent de propager ces opérations aux entités associées :

#[ORM\OneToMany(targetEntity: Feature::class, mappedBy: "product",
    cascade: ["persist", "remove"], // Propage persist() et remove() aux Feature
    orphanRemoval: true             // Supprime une Feature retirée de la collection
)]
private Collection $features;
  • cascade: ["persist"] : persister le Product et ses Feature
  • cascade: ["remove"] : supprimer le Product et ses Feature
  • orphanRemoval: true : une Feature retirée de la collection (donc « orpheline ») est automatiquement supprimée

⚠️ cascade: ["remove"] agit au niveau PHP (Doctrine charge puis supprime chaque entité). Pour déléguer à la base, préférer une contrainte onDelete: "CASCADE" sur la JoinColumn.

Cascade Operations

Chargement : LAZY vs EAGER

L'option fetch définit quand les entités associées sont chargées depuis la base :

Mode Comportement
LAZY (défaut) L'association est chargée à la demande, lors du premier accès (via un proxy)
EAGER L'association est toujours chargée en même temps que l'entité principale
EXTRA_LAZY Permet des opérations (count(), contains()...) sans charger toute la collection 
#[ORM\ManyToOne(targetEntity: Product::class, fetch: "EAGER")]
private ?Product $product = null;

⚠️ Le mode LAZY peut provoquer le problème N+1 requêtes (une requête par entité chargée dans une boucle). La solution recommandée est de faire une jointure explicite (JOIN) dans le QueryBuilder plutôt que de passer en EAGER.

Lazy Loading

Héritage avec Doctrine

Doctrine permet de faire correspondre une hiérarchie de classes PHP à la base de données. Deux stratégies principales existent :

Stratégie Principe Compromis
Single Table Toutes les classes dans une seule table, distinguées par une colonne discriminante Rapide (pas de jointure) mais colonnes des sous-classes nullables
Joined Table Une table par classe, reliées par jointures Base normalisée mais lectures plus lentes (jointures)

Une troisième approche, Mapped Superclass, permet de factoriser des propriétés communes dans une classe parente qui n'est pas elle-même une entité (pas de table dédiée).

Inheritance Mapping

Héritage : Single Table

Une colonne discriminante (type) indique la classe concrète de chaque ligne :

#[ORM\Entity]
#[ORM\InheritanceType("SINGLE_TABLE")]
#[ORM\DiscriminatorColumn(name: "type", type: "string")]
#[ORM\DiscriminatorMap([
    "person" => Person::class,
    "employee" => Employee::class,
])]
class Person
{
    #[ORM\Id]
    #[ORM\GeneratedValue]
    #[ORM\Column]
    private ?int $id = null;

    #[ORM\Column(length: 255)]
    private ?string $name = null;
}
#[ORM\Entity]
class Employee extends Person
{
    #[ORM\Column]
    private ?int $salary = null;
}

La table person contiendra les colonnes id, type, name et salary (nullable pour les lignes de type person).

Renseigner les dates de création / MàJ

Dans une base de données, il est souvent nécessaire d’enregistrer les dates de création ou de mise à jour d'une entité.

Doctrine déclenche automatiquement des événements lorsqu’une action est effectuée sur une entité. La liste de ces actions (prePersist, preRemove, postUpdate...) est disponible dans la documentation de Doctrine.

Ajouter

Code à ajouter dans un Controller :

// L'Entity Manager est récupéré par injection de dépendance
public function testAction(EntityManagerInterface $em): Response
{
    // Création d’un nouveau produit
    $product = new Product();
    $product->setName('Keyboard');

    // Ajouter l’objet dans l’Entity Manager
    $em->persist($product);

    // Exécution des requêtes SQL
    $em->flush();
}

Modifier

Code à ajouter dans un Controller :

// Récupérer le Repository de l'entité
$repository = $em->getRepository(Product::class);

// Récupérer le produit id = 3
$product = $repository->find(3);

// Modification du produit
$product->setName('Monitor');

// Execution des requêtes SQL
$em->flush();

Supprimer

Code à ajouter dans un Controller :

// Récupérer le Repository de l'entité
$repository = $em->getRepository(Product::class);

// Récupérer le produit id = 3
$product = $repository->find(3);

// Suppression du produit
$em->remove($product);

// Execution des requêtes SQL
$em->flush();

Rechercher des données

Doctrine met à disposition des méthodes permettant de récupérer des données de la base de données sans écrire de requête :

  • find()
  • findAll()
  • findBy()
  • findOneBy()

Il existe également des méthodes dynamiques comme par exemple findOneByName(), findByPrice()...

DQL et QueryBuilder

DQL

$em->createQuery('
                SELECT p
                FROM App\Entity\Post p
                WHERE p.publishedAt <= :now
                ORDER BY p.publishedAt DESC
            ')
            ->setParameter('now', new \DateTimeImmutable());

QueryBuilder

$qb = $em->createQueryBuilder();


$qb->select('p')
    ->from(Post::class, 'p')
    ->where($qb->expr()->lte('p.publishedAt', ':now'))
    ->orderBy('p.publishedAt', 'DESC')
    ->setParameter('now', new \DateTimeImmutable());

Custom Repository

Pour créer des requêtes plus complexes, nous pouvons créer des Custom Repositories.

Un Custom Repository est une classe PHP qui nous permettra de stocker toutes les requêtes relatives à une entité et qui ne sont pas auto-générées par Doctrine (find, findAll, findBy…).
