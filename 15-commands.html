Commandes console

Pourquoi créer ses commandes ?

Au-delà des commandes fournies par Symfony (cache:clear, make:*...), on crée souvent ses propres commandes pour les tâches exécutées hors du contexte web :

  • Imports / exports de données
  • Tâches planifiées (CRON) : nettoyage, rappels par e-mail...
  • Traitements de maintenance ou de migration ponctuels

Une commande a accès à tous les services de l'application (Doctrine, Mailer...).

Console

Générer une commande

php bin/console make:command app:send-reminders

Cela crée une classe dans src/Command/. Une commande est identifiée par un nom (convention préfixe:action) déclaré via l'attribut #[AsCommand].

La commande est ensuite listée par :

php bin/console list

Structure d'une commande

namespace App\Command;

use Symfony\Component\Console\Attribute\AsCommand;
use Symfony\Component\Console\Command\Command;
use Symfony\Component\Console\Style\SymfonyStyle;

#[AsCommand(
    name: 'app:send-reminders',
    description: 'Envoie les rappels aux utilisateurs',
)]
class SendRemindersCommand
{
    // Commande invocable : toute la logique tient dans __invoke()
    public function __invoke(SymfonyStyle $io): int
    {
        $io->title('Envoi des rappels');
        // ... logique métier
        $io->success('Rappels envoyés !');

        return Command::SUCCESS;
    }
}

Le style invocable est celui recommandé par la documentation : la classe n'hérite plus de Command (importée uniquement pour ses constantes) et __invoke() renvoie un code de sortie.

Les codes de retour

Le code renvoyé indique au système (et au CRON) si la commande a réussi :

  • Command::SUCCESS : code 0 (succès)
  • Command::FAILURE : code 1 (échec)
  • Command::INVALID : code 2 (mauvaise utilisation)

⚠️ Une commande qui n'échoue jamais explicitement masque ses erreurs : il faut renvoyer FAILURE en cas de problème.

Arguments et options

Une commande accepte des arguments (positionnels) et des options (préfixées par --). En style invocable, ils sont déclarés par des attributs sur les paramètres de __invoke() :

use Symfony\Component\Console\Attribute\Argument;
use Symfony\Component\Console\Attribute\Option;

public function __invoke(
    SymfonyStyle $io,
    #[Argument] string $email,                  // Argument requis
    #[Option('dry-run')] bool $dryRun = false,  // Option : --dry-run
): int {
    $io->text($email);

    if ($dryRun) {
        // ... simuler sans envoyer
    }

    return Command::SUCCESS;
}
php bin/console app:send-reminders john@example.com --dry-run

Un paramètre sans valeur par défaut est requis ; une valeur par défaut le rend facultatif.

Injecter des services

Comme tout service, une commande reçoit ses dépendances par le constructeur :

#[AsCommand(name: 'app:send-reminders')]
class SendRemindersCommand
{
    public function __construct(
        private readonly UserRepository $userRepository,
        private readonly MailerInterface $mailer,
    ) {
    }

    public function __invoke(SymfonyStyle $io): int
    {
        foreach ($this->userRepository->findInactive() as $user) {
            // ... envoyer un rappel
        }

        return Command::SUCCESS;
    }
}

💡 Le style invocable n'héritant pas de Command, il n'y a aucun parent::__construct() à appeler : le constructeur ne sert qu'à l'injection.

SymfonyStyle

La classe SymfonyStyle offre une sortie soignée et cohérente. Reçue en paramètre de __invoke(SymfonyStyle $io), elle expose notamment :

$io->title('Traitement');
$io->section('Étape 1');
$io->text('Message informatif');
$io->success('Terminé avec succès');
$io->warning('Attention...');
$io->error('Une erreur est survenue');

// Interaction
$name = $io->ask('Quel est votre nom ?');
if ($io->confirm('Continuer ?')) { /* ... */ }

// Barre de progression
$io->progressStart(100);
$io->progressAdvance();
$io->progressFinish();

Exercice

  1. Créer une commande app:users:count affichant le nombre d'utilisateurs (injection du repository)
  2. Ajouter un argument role optionnel pour filtrer par rôle
  3. Ajouter une option --format (text ou json) pour le format de sortie
  4. Utiliser SymfonyStyle (titre, tableau, message de succès)
  5. Renvoyer FAILURE si le rôle demandé n'existe pas
