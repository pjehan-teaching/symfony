API & Sérialisation

Exposer une API

Une API REST renvoie des données structurées (généralement en JSON) plutôt que du HTML, afin d'être consommée par une application front (JavaScript, mobile...) ou un autre service.

Le cœur du problème est la sérialisation : transformer un objet PHP (une entité Doctrine) en JSON, et inversement.

Serializer

Installation

composer require symfony/serializer-pack

Une fois le composant installé, la méthode $this->json() du contrôleur l'utilise automatiquement (au lieu d'un simple json_encode()) : 

#[Route('/api/products', name: 'api_product_list', methods: ['GET'])]
public function list(ProductRepository $repository): Response
{
    $products = $repository->findAll();

    // Les entités sont sérialisées en JSON par le Serializer
    return $this->json($products);
}

Le service Serializer

Pour un contrôle plus fin, on injecte SerializerInterface :

Objet → JSON

$json = $serializer->serialize(
    $product,
    'json'
);

JSON → Objet

$product = $serializer->deserialize(
    $request->getContent(),
    Product::class,
    'json'
);

En interne, le Serializer combine des normalizers (objet ↔ tableau) et des encoders (tableau ↔ JSON / XML / CSV).

Les groupes de sérialisation

Par défaut, toutes les propriétés sont exposées. C'est un problème : on ne veut pas renvoyer le mot de passe d'un utilisateur, ni provoquer des références circulaires entre entités liées.

Les groupes permettent de choisir les propriétés exposées selon le contexte :

use Symfony\Component\Serializer\Attribute\Groups;

class Product
{
    #[Groups(['product:read', 'product:write'])]
    private ?string $name = null;

    #[Groups(['product:read'])]
    private ?int $id = null;

    // Aucun groupe : jamais sérialisé via un groupe
    private ?string $internalNote = null;
}
$json = $serializer->serialize($product, 'json', ['groups' => 'product:read']);

Recevoir des données : MapRequestPayload

Pour désérialiser automatiquement le corps JSON d'une requête en objet, Symfony fournit l'attribut #[MapRequestPayload] : 

use Symfony\Component\HttpKernel\Attribute\MapRequestPayload;

#[Route('/api/products', name: 'api_product_create', methods: ['POST'])]
public function create(
    #[MapRequestPayload] ProductDto $dto,
    EntityManagerInterface $em
): Response {
    // $dto est désérialisé depuis le JSON ET validé automatiquement
    $product = new Product();
    $product->setName($dto->name);

    $em->persist($product);
    $em->flush();

    return $this->json($product, Response::HTTP_CREATED);
}

💡 On désérialise vers un DTO (Data Transfer Object) plutôt que directement vers l'entité, afin de ne pas exposer sa structure interne et de valider les données entrantes.

Aller plus loin : API Platform

Pour une API complète, écrire manuellement chaque endpoint (liste, détail, création, modification, suppression, pagination, filtres...) devient vite répétitif.

API Platform est un framework bâti sur Symfony qui génère une API REST (et GraphQL) à partir de simples attributs sur les entités, avec documentation OpenAPI / Swagger automatique. 

use ApiPlatform\Metadata\ApiResource;

#[ApiResource]
class Product
{
    // Les opérations CRUD REST sont exposées automatiquement
}
